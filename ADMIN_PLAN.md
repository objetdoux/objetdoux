# objetdoux Admin Page Plan

## 1. 관리자 페이지 목적

오브제두 관리자 페이지는 초기 오픈 단계에서 다음 업무를 처리하기 위한 내부 운영 도구입니다.

- 상품 등록 및 수정
- 상품 노출/품절 관리
- 주문 확인
- 주문 상태 변경
- 회원 기본 정보 확인
- 메인 배너 및 일부 콘텐츠 관리 여부 검토

초기 버전에서는 모든 기능을 크게 만들기보다, 실제 운영에 꼭 필요한 `상품 관리`와 `주문 관리`를 중심으로 구축합니다.

## 2. 관리자 페이지 기본 방향

- 경로: `/admin`
- 관리자 로그인 필요
- 일반 사용자 페이지와 분리
- 디자인 톤은 사용자 페이지보다 더 실용적으로 구성
- 브랜드 톤은 유지하되, 관리 효율을 우선
- 모바일 대응은 최소한으로 하되, PC 운영을 기준으로 설계

## 3. 1차 관리자 MVP 범위

### 반드시 포함

- 관리자 로그인
- 관리자 대시보드
- 상품 목록
- 상품 등록
- 상품 수정
- 상품 노출/비노출
- 품절 여부 관리
- 주문 목록
- 주문 상세
- 주문 상태 변경
- 회원 목록

### 있으면 좋지만 1차 이후 가능

- 메인 배너 관리
- 브랜드/소셜/매거진 콘텐츠 관리
- 상품 이미지 순서 관리 고도화
- 주문 검색 고도화
- 엑셀 다운로드
- 리뷰 관리
- 쿠폰/포인트 관리

### 1차에서 제외

- 복잡한 권한 관리
- 여러 관리자 역할
- 통계 대시보드 고도화
- 자동 배송 연동
- 교환/반품 자동화
- 재고 입출고 관리

## 8. Members 회원 관리

초기 회원 관리는 조회 중심으로 시작합니다.

### 회원 목록 페이지

경로: `/admin/members`

현재 구현:

- Supabase `users` 테이블 기준 회원 목록 조회
- Supabase `orders` 테이블 기준 주문 수, 최근 주문일, 누적 주문 금액 집계
- 취소 주문은 회원별 주문 수와 누적 주문 금액에서 제외
- 검색어와 정렬은 URL 검색 파라미터 기반으로 적용

#### 표시 컬럼

- 이름
- 이메일
- 연락처
- 가입일
- 로그인 방식
- 주문 수
- 최근 주문일
- 누적 주문 금액
- 계정 상태

#### 검색

- 이름
- 이메일
- 연락처

#### 정렬

- 최근 가입순

### 회원 상세는 1차에서 선택

추후 필요 시:

- 회원 주문 내역
- 배송지 목록
- 관심 상품
- 계정 상태

## 9. Settings 기본 설정

경로: `/admin/settings`

초기에는 큰 기능보다 운영에 필요한 기본값을 정리하는 화면으로 둡니다.

현재 구현:

- Supabase `site_settings` 테이블 기준 설정값 조회
- 값이 없을 때는 기본값 표시
- 저장 시 key/value 형태로 upsert
- 저장 후 `/admin/settings` 재검증
- 메인 배너 문구와 EVENT 문구를 저장하고 메인 화면에 반영
- 메인 배너 이미지와 EVENT 이미지를 Storage에 업로드하고 메인 화면에 반영
- 저장한 배송비, 무료 배송 기준, 배송 안내는 장바구니/주문서/주문 생성에 사용
- 품절/비노출 상품은 주문 생성 전 서버에서 다시 검증

### 설정 항목

- 메인 배너 제목
- 메인 배너 설명
- 메인 배너 이미지
  권장: 1920 x 720px 이상, 가로형
- 이벤트 제목/설명
- 이벤트 이미지
  권장: 1200 x 700px 이상, 가로형
- 기본 배송비
- 무료 배송 기준
- 제주/도서산간 추가 배송비 안내
- 고객센터 이메일
- 고객센터 연락처
- 사업자 정보
- 반품/교환 안내 문구

1차에서는 DB 저장 기능 없이 정적 화면으로 시작해도 됩니다.

## 10. 관리자 권한

초기에는 관리자 계정을 1개 또는 소수만 운영하는 전제로 갑니다.

### 현재 구현

- `/admin/login` 화면 존재
- Supabase Auth 이메일/비밀번호 로그인
- `admin_users`에 등록된 활성 이메일만 관리자 접근 허용
- `/admin` 이하 화면은 `AdminShell`에서 서버 사이드로 관리자 권한 확인
- 로그아웃 버튼 제공

추후 운영 전에는 로그인 시도 제한, 2단계 인증, 권한별 접근 제어를 추가 검토합니다.

### 1차 권한 구조

- `admin`
  모든 관리자 기능 접근 가능

### 나중에 확장 가능

- `owner`
- `manager`
- `staff`

하지만 초기에는 권한을 나누지 않는 것을 추천합니다.

## 11. 데이터 구조 관점

관리자 페이지가 다룰 핵심 테이블은 다음과 같습니다.

- `users`
- `products`
- `product_images`
- `categories`
- `orders`
- `order_items`
- `addresses`
- `wishlist_items`

초기에는 카테고리를 별도 테이블로 만들지 않고 문자열로 관리해도 됩니다.

다만 추후 카테고리 관리가 필요하다면 `categories` 테이블을 분리합니다.

## 12. 상품 데이터 필드 초안

```text
products
- id
- slug
- name
- category
- price
- summary
- description
- material
- size
- track_stock
- stock_quantity
- thumbnail_url
- image_urls
- detail_image_url
- is_visible
- is_sold_out
- is_new
- created_at
- updated_at
```

재고 관리 기준:

- `track_stock`을 켠 상품은 관리자에서 입력한 `stock_quantity`를 주문 가능 수량으로 사용합니다.
- 장바구니 담기, 장바구니 수량 변경, 주문 생성 전 단계에서 재고 부족 여부를 확인합니다.
- 주문 생성 후 DB 함수로 상품 행을 잠근 뒤 주문 수량만큼 재고를 차감하고, 재고가 0개가 되면 `is_sold_out`을 자동으로 켭니다.

## 16. 초기 관리자 MVP 결론

오브제두 초기 관리자 페이지는 크게 두 가지에 집중합니다.

1. 상품을 등록하고 수정할 수 있다.
2. 주문을 확인하고 상태를 바꿀 수 있다.

회원 관리와 설정은 최소 조회/수정 수준으로만 두고, 리뷰/쿠폰/통계/자동화는 오픈 이후 운영 상황을 보고 추가합니다.
